# Open Responses Provider Feature Parity

Sources used: Open Responses specification site and compatible provider references; local AI SDK OpenAI Responses implementation as the closest mature Responses-shaped adapter.

| Feature | Provider docs behavior | AI SDK behavior | SwiftAgent current behavior | Gap | Priority | Tests needed |
| --- | --- | --- | --- | --- | --- | --- |
| Responses-compatible endpoint | Open Responses defines a shared Responses-style schema for multi-provider inference. | AI SDK OpenAI adapter is Responses-shaped but OpenAI-specific. | `OpenResponsesLanguageModel` posts to configurable `baseURL` using `responses`. | Need clarify which OpenAI-only extensions are intentionally supported. | High | Provider-agnostic request fixture. |
| Text generation | Returns output text/items. | Supported by Responses implementation. | Supported. | None for basic text. | Done | Existing fixture. |
| Streaming text | Spec centers streaming events/items. | Typed stream parts. | Supports text deltas from Responses-style events. | Needs acceptance against Open Responses spec events, not only OpenAI-shaped events. | High | Open Responses SSE fixture. |
| Structured output | Uses Responses-style text format/schema. | Supported. | Supported with JSON schema format. | Strict/provider-specific behavior not fully audited. | Medium | Structured request/response fixture. |
| Image input | Responses-style multimodal input items. | Supported for OpenAI. | Supported as input images. | Need Open Responses acceptance check for URLs/base64 fields. | Medium | Multimodal request fixture. |
| Local function tools | Function tools produce function-call output items. | Supported for OpenAI. | Supported non-streaming and streaming. | Need spec-level request/replay tests. | High | Function call/output replay. |
| Tool choice/allowed tools | Spec includes `tool_choice`, allowed tools, and max tool calls. | Supports OpenAI variants. | Supports tool choice, allowed tools, parallel tool calls, max tool calls. | Need spec conformance for allowed-tools wire shape. | Medium | Request-body tests. |
| Reasoning items | Open Responses formalizes reasoning content, summary, and encrypted content. | OpenAI adapter preserves reasoning/encrypted content. | Parses summary/encrypted content when present. | Need support audit for raw reasoning `content` and item references. | High | Reasoning fixture with content/summary/encrypted. |
| `store` | Compatible providers may implement stored response behavior. | OpenAI adapter defaults stored behavior for OpenAI. | Option exists and is sent when set. | Need document provider variability; no continuation policy. | Medium | Request test. |
| Response continuation | Compatible providers may support item replay and/or stored IDs. | OpenAI adapter supports item replay plus provider options. | Replays known response items through provider metadata. | No first-class previous-response/conversation equivalent in this adapter. | High | Multi-turn request fixture. |
| `include` extensions | Some providers expose include fields such as encrypted reasoning, file-search results, code outputs. | OpenAI adapter has typed include. | Supported through `OpenResponsesLanguageModel.CustomGenerationOptions.include`. | Provider-specific include names are caller supplied strings. | Done | Request test added. |
| Hosted tools | Spec allows shared and provider-specific tool surfaces. | AI SDK models OpenAI hosted tools. | Raw provider-defined request support added through `providerTools` and `OpenResponsesLanguageModel.ProviderTool.raw`. Provider-owned tool calls pass through without local execution. | Typed provider-specific helpers and full result normalization remain follow-up. | Medium | Request test added; hosted tool fixture still needed. |
| Citations/annotations/logprobs | Provider-specific outputs can attach metadata. | AI SDK maps OpenAI annotations/source metadata. | Not fully normalized. | Missing provider metadata matrix and decoding coverage. | Medium | Decode fixture. |
| Usage/HTTP metadata | Responses-style APIs return usage and response metadata. | Preserved. | Preserved where returned. | Needs latest provider header audit. | Low | Header/usage fixture. |
